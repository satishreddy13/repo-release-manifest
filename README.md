# Release Manifest Repository

Owned by CM. This repository is the single source of truth for every bi-weekly
sprint release. It does **not** contain application code — it records which
branch/commit or SharePoint artifact version from each team is in a given build,
what COTS version is included, and the deployment status of each environment.

---

## Team Delivery Models

Teams do not all need to use Git. The manifest supports two delivery models:

| Model | Teams | How CM tracks | Tagging |
|-------|-------|---------------|---------|
| **Git** | Interfaces, Workflow Config, Func Config | Branch + commit SHA | `sit/`, `uat/`, `release/` git tags |
| **SharePoint** | Conversion | SharePoint folder URL + artifact version (e.g. `v1.3`) | Artifact version recorded in manifest |

Both models are first-class — any team can be included, deferred, or removed
using the same process regardless of delivery platform.

---

## Cross-Team Dependency Tracking

The Teams sheet has a `depends_on` column (comma-separated `team_key` values).
When `generate-report.sh` runs, it warns if any depended-on team is deferred:

```
⚠️  DEPENDENCY WARNINGS:
   Interfaces depends on Functional Product Config — but Functional Product Config is DEFERRED
```

The warning also appears as a red band in `RELEASE-NOTES.xlsx`. CM must review
and decide whether to defer the dependent team before cutting the build.

For fuller detail on the options considered see
[`docs/dependency-management-options.md`](docs/dependency-management-options.md).

## Repository Layout

```
releases/
  sprint-NN/
    manifest.xlsx         ← build definition (CM edits this — Excel file)
    RELEASE-NOTES.md      ← auto-generated human-readable report
    cots-sprint-NN.md     ← COTS version record for this sprint

templates/
  manifest.xlsx.template  ← generated by create_manifest_xlsx.py at sprint start
  cots.md.template        ← copy and fill in COTS details each sprint

scripts/
  cut-sprint.sh            ← CM runs at sprint START  — creates git branches + notifies SharePoint teams
  create_manifest_xlsx.py  ← called by cut-sprint.sh  — generates the Excel manifest
  generate-report.sh       ← CM runs at BUILD DAY     — produces RELEASE-NOTES.md
  generate-report.py       ← called by generate-report.sh — reads manifest.xlsx
  tag-environments.sh      ← CM runs at DEPLOYMENT    — stamps sit/uat tags (git) + records artifact version (SharePoint)
  close-sprint.sh          ← CM runs after UAT passes — merges to main, tags release (git teams only)
```

## CM Sprint Runbook

### Sprint Start
```bash
./scripts/cut-sprint.sh sprint-13
```
- **Git teams**: creates `sprint/sprint-13` branch in each repo and pushes to GitHub
- **SharePoint teams**: prints the expected SharePoint folder URL and a reminder to notify the team
- Scaffolds `releases/sprint-13/manifest.xlsx` (open in Excel and fill in team details)
- Scaffolds `releases/sprint-13/cots-sprint-13.md`

### Build Day

1. **SharePoint teams** confirm their deliverable folder/version is ready (e.g. `v1.3`)
2. **Git teams** confirm their sprint branch is ready; CM records the HEAD commit SHA
3. CM opens `releases/sprint-NN/manifest.xlsx`:
   - **Teams sheet**: for each team set `included` = TRUE/FALSE, fill in `commit_or_artifact_version`; if the team depends on another team's work being in the build, fill in `depends_on` (e.g. `func_config`)
   - **COTS sheet**: update version, hotfixes
   - **Build Info sheet**: fill in `build_date`, `built_by`
4. CM runs:
   ```bash
   ./scripts/generate-report.sh sprint-12
   ```
   Writes `releases/sprint-12/RELEASE-NOTES.xlsx`. If any included team's `depends_on` team is deferred, a warning is printed to the terminal and shown in the report — CM must review before proceeding.
5. CM commits manifest + report, opens PR titled `Release sprint-12`.
6. Deploy included teams to SIT, then:
   ```bash
   ./scripts/tag-environments.sh sprint-12 sit
   ```
   - Git teams: `sit/sprint-12` tag created on the recorded commit SHA
   - SharePoint teams: artifact version confirmed; no git tag needed

### SIT Passes → Promote to UAT
```bash
./scripts/tag-environments.sh sprint-12 uat
```

### UAT Passes → Close Sprint
```bash
./scripts/close-sprint.sh sprint-12
```
- **Git teams**: merges sprint branch → `main`, creates `release/sprint-12` tags
- **SharePoint teams**: confirms artifact version is the release record (no merge needed)
- Updates manifest: `environments.uat.status = passed`

## Removing a Team Mid-Sprint

**Before build is cut:** Open `manifest.xlsx`, set the team's `included` = FALSE and fill in `reason_if_deferred`. No branch or folder changes needed.

**After deployment to SIT:**
```bash
cp -r releases/sprint-12 releases/sprint-12-patch1
# Open releases/sprint-12-patch1/manifest.xlsx
# Set problem team to included: FALSE with reason
./scripts/generate-report.sh sprint-12-patch1
./scripts/tag-environments.sh sprint-12-patch1 sit
```
The original `sprint-12` manifest is preserved as an immutable audit record.

## Branch Naming Conventions (Git teams)

| Branch | Who creates | Purpose |
|--------|-------------|---------|
| `main` | CM | Stable — tagged releases only |
| `sprint/sprint-NN` | CM | Sprint integration branch |
| `feature/ELM-XXXX-short-desc` | Developer | Individual work item |
| `fix/ELM-XXXX-short-desc` | Developer | Bug fix or hotfix |

## Commit Message Convention (Git teams)
```
ELM-1234: Short description of what changed

Optional longer body explaining why.
```

## Environment Tag Convention (Git teams)

| Tag | Meaning |
|-----|---------|
| `sit/sprint-NN` | Exact commit deployed to SIT |
| `uat/sprint-NN` | Exact commit deployed to UAT |
| `release/sprint-NN` | Commit merged to main after UAT passed |

## SharePoint Artifact Versioning Convention

SharePoint teams record their deliverable version in the manifest's
`commit_or_artifact_version` field. Recommended format: `v<major>.<minor>`
(e.g. `v1.3`). The folder URL in `repo_or_sharepoint_url` should point to
the sprint-specific folder containing the final deliverables.
